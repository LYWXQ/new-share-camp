# 功能完善与验证

## 功能
1. **需求分析与完善**
   - 分析模糊的功能描述
   - 结合相关代码和文档理解需求
   - 识别缺失的功能细节
   - 生成完善后的功能需求

2. **用户确认流程**
   - 在执行前向用户展示补充的内容
   - 确认功能范围和实现方案
   - 记录用户的确认决策

3. **范围分析**
   - 确定功能涉及的范围：
     - 仅前端修改
     - 仅后端修改
     - 仅数据库修改
     - 前端+后端+数据库全链路
     - 其他组合

4. **流程验证**
   - 检查API是否完成对接
   - 验证前端到后端到数据库整个流程
   - 确保数据流的完整性

## 使用场景
- 当用户要求进行功能修改、删除、增加时
- 当功能描述模糊或不完整时
- 当需要确认功能实现范围时
- 当需要验证全栈流程时

## 工作流程

### 1. 需求分析阶段
1. 接收用户的功能需求
2. 检查需求描述是否模糊
3. 搜索相关代码和文档
4. 分析现有代码结构
5. 识别功能的技术实现方式

### 2. 需求完善阶段
1. 列出缺失的关键信息
2. 基于代码分析补充功能细节
3. 考虑可能的边界条件
4. 生成完整的功能说明

### 3. 用户确认阶段
1. 向用户展示补充的内容
2. 明确说明修改了什么
3. 等待用户确认
4. 只有确认后才继续执行

### 4. 范围确定阶段
1. 分析功能涉及的技术层次
2. 检查前端组件
3. 检查后端API
4. 检查数据库结构
5. 确定修改范围

### 5. 流程验证阶段
1. 检查API接口是否存在
2. 验证API参数和返回值
3. 检查前端调用方式
4. 验证数据库操作
5. 确保整个链路畅通

### 6. 执行阶段
1. 根据确认的需求进行开发
2. 遵循确定的修改范围
3. 保持代码风格一致
4. 进行充分的测试

## 关键检查点

### 前端检查
- [ ] 相关页面/组件是否存在
- [ ] UI交互逻辑是否完整
- [ ] 状态管理是否需要更新
- [ ] API调用是否正确

### 后端检查
- [ ] API接口是否已实现
- [ ] 业务逻辑是否完整
- [ ] 数据验证是否到位
- [ ] 错误处理是否完善

### 数据库检查
- [ ] 表结构是否需要变更
- [ ] 索引是否需要优化
- [ ] 数据迁移是否需要
- [ ] 数据完整性约束

### 流程检查
- [ ] 前端→后端API对接
- [ ] 后端→数据库数据操作
- [ ] 数据返回和前端展示
- [ ] 错误信息传递链路

## 示例

### 场景：用户要求"添加用户删除功能"
1. **分析现有代码**：查找用户管理相关代码
2. **完善需求**：
   - 补充：删除前需要确认
   - 补充：删除后需要刷新列表
   - 补充：软删除还是硬删除
3. **向用户确认**：展示补充的需求细节
4. **确定范围**：前端(确认对话框+列表刷新) + 后端(删除API) + 数据库(删除记录)
5. **验证流程**：检查API是否存在，确认数据库操作
6. **执行开发**

## 注意事项
- 始终在完善需求后向用户确认
- 不要假设用户的意图
- 充分考虑技术实现的复杂性
- 确保每个修改环节都经过验证
